Detection, Audit Logging & the “Single Tool” Question

Detection starts from the assumption that, despite every control described in Sections 3 through 9, something will eventually get through, its job is to notice quickly, not to prevent in the first place. Prevention is never a 100% guarantee, especially in an open-source stack where new vulnerabilities surface continuously; detection is what turns an undiscovered breach into a contained, handled incident instead of an ongoing, unnoticed one.

This is deliberately kept as its own dedicated security pipeline, separate from the infrastructure monitoring stack (Prometheus, Grafana, Alertmanager) documented separately for this deployment. auditd tracks system calls at the OS level on every node, covering file integrity, unusual access patterns, and privilege escalation attempts, with its own dedicated alert path rather than being bolted onto infrastructure metrics. Keystone, Nova, and Neutron audit events are shipped centrally through OpenSearch so a complete who-did-what-when record exists independently of any other tool.

Control

Implementation

Host intrusion detection

auditd deployed on every node with rules watching file integrity, unusual access patterns, and privilege escalation attempts, reporting to a central log store.

System call auditing

auditd enabled on every node (auditctl -w /etc/kolla -p wa -k kolla_config_changes) tracking file access and privilege escalation at the OS level.

OpenStack audit trail

Keystone, Nova, and Neutron CADF audit events shipped through the Fluentd pipeline into OpenSearch, giving a full who-did-what-when record.

Dedicated security alerting

Security-specific alerts (e.g. a spike in failed Keystone logins) are routed through their own notification channel, kept distinct from infrastructure alerting so security events are never diluted among routine capacity or performance alerts.

Is There a Single Tool That Secures Everything?

No single product performs authentication, network firewalling, container scanning, host hardening, and intrusion detection natively and well, these are fundamentally different kinds of controls, not variations of one underlying idea. This is not a limitation specific to OpenStack; it holds true across virtually every real IT environment.

The closest realistic equivalent to a single pane of glass is a tool that all of the above signals feed into and get correlated by, rather than a tool that performs every control itself. Vitrage, an official OpenStack project built for root-cause correlation, is the OpenStack-native version of that idea: instead of securing anything itself, it correlates alerts already generated by the controls in Sections 3 through 9, so a single Ceph OSD failure that would otherwise trigger twenty separate alerts across Nova, Cinder, and Ceph is presented as one incident with one root cause, not twenty unrelated ones.

Common pitfalls

  • Deploying auditd with overly broad rules, resulting in alert volume too high for anyone to realistically review.

  • Treating the presence of audit logs as equivalent to actually monitoring them.

  • Conflating security detection with infrastructure monitoring so closely that a genuine security alert gets lost among routine capacity warnings.
